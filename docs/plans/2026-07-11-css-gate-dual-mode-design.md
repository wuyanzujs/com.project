# CSS 双门禁设计

生成日期：2026-07-11

适用范围：`apps/com.deppon.app`

## 背景

项目当前只完成约 30% 的重构，后续将对现有业务样式进行全量重写。门禁既不能继续放任旧 CSS 增长，也不能在治理开始前让整个项目的日常验证永久失败。

## 决策

采用双门禁：

1. 日常门禁 `check:styles` 保持当前项目可验证，但把存量债务从“全项目总额”改为“逐文件精确基线”。每个旧文件只能减少颜色、字号、行高、圆角、字重和体量，不能通过清理另一个文件来抵消新增债务。
2. 严格门禁 `check:styles:strict` 表示 CSS 全量治理的最终完成条件。它要求所有业务 SCSS 使用 token、视觉字面量归零、文件体量达标、TS/TSX 静态颜色归零，并消除子组件导入父级 `index.scss`。
3. 基线更新命令只允许记录债务下降、新增零债务文件和已删除文件。任何指标上升都拒绝写入，避免为了通过 CI 人为抬高上限。

CSS 治理完成后，严格门禁进入主 `verify`，逐文件存量基线退出历史兼容角色。

## 日常门禁行为

- 已登记文件的每个指标不得超过自身基线。
- 指标下降但基线未同步更新时失败，要求在同一次变更中收紧基线，避免产生新的回涨空间。
- 新增 SCSS 必须引入 token、视觉字面量为零，并满足页面或组件行数预算。
- 新出现的 TS/TSX 静态颜色直接失败。
- 新增子组件导入父级 `../index.scss` 直接失败。
- 删除或新增文件后必须使用受控命令刷新文件清单。

## 严格门禁行为

- 所有业务 SCSS 必须引入 token。
- 十六进制、颜色函数、命名颜色和 `transparent` 等静态颜色全部归零。
- `font-size`、`line-height`、`border-radius`、`font-weight` 静态值全部归零。
- 页面 SCSS 不超过 300 行；页面子组件和共享组件 SCSS 不超过 180 行。
- `src/styles/nativeTokens.ts` 之外不保留 TS/TSX 静态颜色。
- 子组件不再导入父页面 `index.scss`。
- 不再保留 `.dp-*` 全局类原型。

## 基线数据

基线使用版本化 JSON，保存：

- 每个业务 SCSS 的视觉字面量、token 接入状态和行数。
- 每个 TS/TSX 文件的静态颜色数量。
- 父样式导入和 legacy 全局类文件清单。
- 最终严格目标和页面/组件行数预算。

## 验证

- 使用 Node 内置测试验证颜色识别、逐文件回涨、基线收紧、新文件规则和严格模式。
- 日常验证运行测试、Stylelint、日常样式门禁和 RN 边界检查。
- 严格验证单独运行，并在治理完成前输出剩余债务，不进入日常 `verify`。

## 不在本次范围

- 不重写任何页面 SCSS。
- 不修改现有页面视觉或 TSX 结构。
- 不在本次将严格门禁加入主 `verify`。
