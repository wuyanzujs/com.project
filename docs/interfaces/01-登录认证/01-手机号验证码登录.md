# 手机号验证码登录

返回 [[01-登录认证/00-模块总览|登录认证接口]]，也可以回到 [[00-接口总览]]。

## 接口说明

| 项目      | 说明                                                        |
| --------- | ----------------------------------------------------------- |
| 用途      | 手机号验证码登录，登录成功后返回用户信息并建立 App 登录态。 |
| 源码方法  | `authApi.login`                                             |
| 源码位置  | `apps/com.deppon.app/src/services/auth/auth.api.ts:11`      |
| HTTP 方法 | `POST`                                                      |
| 接口路径  | `/gwapi/userService/eco/user/login`                         |
| 登录态    | 否，登录接口自身不要求已有会话                              |
| 请求类型  | `LoginRequest`                                              |
| 响应类型  | `DepponResponse<AppUser>`（前端归一）                       |

## 请求字段

类型：`LoginRequest`。

| 字段         | 必填 | 类型     | 说明                                                |
| ------------ | ---- | -------- | --------------------------------------------------- |
| `account`    | 是   | `string` | 登录账号，当前通常传手机号。                        |
| `verifyCode` | 是   | `string` | 短信验证码。                                        |
| `loginType`  | 是   | `string` | 登录方式。                                          |
| `sysCode`    | 是   | `string` | 系统来源编码，用于区分 App、小程序、H5 等调用来源。 |
| `registerRecord` | 是 | `LoginRegisterRecord` | 旧接口要求的注册来源信封；RN 只保留安全的 App 来源页和空兼容字段。 |

## 响应字段

前端统一归一为 `DepponResponse<AppUser>`：业务成功时 `status=true`，用户信息位于 `result`；`message` 保留后端提示，`sessionCookieSaved` 表示本次响应是否已成功建立本地 `ECO_TOKEN` 会话。原始 OWS 包体不会把用户信息直接当作登录凭证。

`result` 类型：`AppUser`。

| 字段                | 必填 | 类型       | 说明                    |
| ------------------- | ---- | ---------- | ----------------------- |
| `id`                | 否   | `string`   | 数据主键或业务记录 ID。 |
| `mobile`            | 否   | `string`   | 手机号。                |
| `mobileEncrypt`     | 否   | `string`   | 加密后的手机号。        |
| `nickName`          | 否   | `string`   | 昵称。                  |
| `openId`            | 否   | `string`   | 微信或渠道 openId。     |
| `originalOpenId`    | 否   | `string`   | 原始 openId。           |
| `unionId`           | 否   | `string`   | 微信 unionId。          |
| `userName`          | 否   | `string`   | 用户名称。              |
| `avatarUrl`         | 否   | `string`   | 头像地址。              |
| `customerCode`      | 否   | `string`   | 客户编码。              |
| `contactCode`       | 否   | `string`   | 联系人编码。            |
| `allUserNames`      | 否   | `string[]` | 关联用户名列表。        |
| `thirdBindUserName` | 否   | `string`   | 第三方绑定用户名。      |

说明：该类型带有索引签名，后端可能返回额外字段，当前文档只展开源码中已声明的字段。

## RN 会话建立

- OWS 登录态仍由响应 cookie 中的 `ECO_TOKEN` 建立，不把用户信息包体当作登录凭证。
- Taro RN `fetch` 不能可靠向 JS 暴露 `Set-Cookie`；`request/cookieJar.ts` 会先尝试兼容响应头，再通过 `shared/native/AppSessionCookie.ts` 从系统 cookie jar 读取并写入异步 storage。
- Android RN 网络层写系统 CookieJar 是异步动作。成功包体已返回但首次读取尚未命中时，登录 service 会通过 cache/request facade 做最多 5 次、每次间隔 40ms 的有界恢复；仍无 `ECO_TOKEN` 才拒绝假登录并清理会话。
- 请求统一使用 `credentials: include`。只有 `ECO_TOKEN` 与用户信息都成功持久化，`authService.loginWithSms` 才返回成功。
- 登录和短信接口标记为 `login=false`，失败响应不会误触发全局 `authExpired` 跳转。

## App 返回约定

- 当前短信登录是 App 内页面流程：业务入口 `navigateTo` 登录页，登录成功后按安全 `redirectUrl` 使用 `replace` 返回，不使用小程序 `switchTab`、H5 私有回调或 URL scheme。
- 支付结果和账号注销同样使用 App route；送货进仓、电子云签等受控 H5 数据回传才使用 `postMessage -> 一次性 bridge -> navigateBack -> useDidShow consume`。
- `auth=Y` 的 WebView 页面会在渲染前再次检查 `ECO_TOKEN`；无会话时保存完整 Web App 路由并走统一登录守卫，成功后再返回原 Web 服务。
